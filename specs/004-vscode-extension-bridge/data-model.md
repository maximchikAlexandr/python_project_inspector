# Data Model: VS Code Extension — Thin Bridge

**Date**: 2026-06-22 | **Spec**: [spec.md](spec.md) | **Plan**: [plan.md](plan.md)

This feature adds no new persistent store. The DuckDB store remains owned by the CLI/worker (Principle V); the extension holds only transient in-process state. The model below defines the **runtime entities** the extension and the new CLI run modes exchange, with their fields, validation, and state transitions. Wire formats are normative in `contracts/`; this document describes the conceptual entities and their in-memory shapes (Python `msgspec` structs, TypeScript interfaces).

## Entity: Analysis Run

A single invocation of `ppi analyze` against a workspace, owned by the extension's `analyzeRunner`.

**Fields**:
- `run_id: str` — UUID generated by the CLI (existing `RunMeta.run_id`).
- `workspace_folder: str` — absolute path of the analyzed folder (FR-017 selection result).
- `profile: "python" | "odoo"` — resolved profile (FR-011).
- `status: "queued" | "running" | "completed" | "failed" | "cancelled"` — run status (FR-020 adds `cancelled`).
- `progress: CommitProgress | None` — last progress event (processed/total/short_hash).
- `error: RunFailure | None` — populated on `failed`.
- `cli_process: Subprocess | None` — the spawned `ppi analyze` process handle (cancellable).

**State transitions**:
```
queued --(spawn ok)--> running --(RunCompleted)--> completed
                          \------(RunFailed)-----> failed
                          \------(cancel/SIGTERM)-> cancelled
```
Validation: at most one `running` run per `workspace_folder` (FR-006); a second launch while `running` is rejected and offers cancel. `cancelled` requires no result rollback (FR-020).

## Entity: Workspace Profile

Per-workspace analysis interpretation plus results-location configuration.

**Fields**:
- `profile: "python" | "odoo"` — default `python` (FR-011).
- `analysis_dir: str | None` — custom results dir, else CLI default (FR-013).
- `python_executable: str | None` — interpreter used to run `ppi` (FR-014).
- `cli_path: str | None` — explicit path to the `ppi` console script (FR-014).
- `scope: "workspace" | "global"` — resolution scope that produced this value (for diagnostics).

**Resolution rule**: VS Code native workspace-over-global (FR-012). Resolution order for the executable: `python_executable` (`<exe> -m ppi`) → `cli_path` → `ppi` on PATH; failure raises `CliNotFound` (FR-014).

## Entity: Dashboard View

The embedded instance of the existing frontend in a Webview panel, bound to a workspace's results.

**Fields**:
- `panel: WebviewPanel` — the VS Code panel handle.
- `workspace_folder: str` — the folder whose results are shown.
- `bridge: QueryBridge` — the `ppi rpc` servant backing this panel (one per panel).
- `state: "empty" | "loading" | "ready" | "stale" | "error"` — panel data state.
- `run_in_progress: bool` — whether an analysis run is active for this folder.

**Lifecycle**: `open → docked(moved) → hidden → revealed → closed`; on close the `QueryBridge` servant is terminated. State `empty` when `status.store_present == false` (FR-010); `stale` when `status.schema_compatible == false`.

## Entity: ProgressEvent (CLI → extension, `analyze --json`)

`msgspec` tagged union emitted as JSON-lines on stdout by `ppi analyze --json`. See `contracts/analyze-json-progress.md` for the exact wire schema.

**Variants**:
- `RunStarted { run_id, branch, mode, commits_total }`
- `CommitProgress { processed, commits_total, short_hash }`
- `RunCompleted { run_id, commits_succeeded, commits_failed, duration_ms }`
- `RunFailed { run_id, exit_reason, message, stderr_tail }`

Validation: `0 <= processed <= commits_total`; `short_hash` is the first 8 chars of the commit hash; events are ordered `RunStarted → CommitProgress* → (RunCompleted | RunFailed)`.

## Entity: RpcRequest / RpcResponse (extension ↔ `ppi rpc`, stdio JSON-RPC)

Read-only query protocol. See `contracts/query-rpc.md`.

**RpcRequest**: `{ id: int, method: str, params: dict }` — `method` mirrors a dashboard `/api` endpoint id; `params` carries query parameters.
**RpcResponse**: `{ id: int, result: dict | list }` on success.
**RpcError**: `{ id: int, error: { code: str, message: str } }` on failure (e.g. `STORE_NOT_FOUND`, `SCHEMA_INCOMPATIBLE`, `QUERY_NOT_FOUND`, `LOCKED`).

Validation: `id` correlates request↔response; unknown `method` → `RpcError` code `METHOD_NOT_FOUND`. The servant is read-only; any write-attempt method is rejected.

## Entity: WebviewMessage (Webview ↔ extension, `postMessage`)

See `contracts/webview-bridge.md`.

**Request** (Webview → ext): `{ kind: "request", id: int, method: str, params: dict }`
**Response** (ext → Webview): `{ kind: "response", id: int, result?: ..., error?: {...} }`
**Command** (Webview → ext): `{ kind: "command", command: "ppi.analyze" | "ppi.openSettings" | "ppi.cancelAnalysis" }`
**Event** (ext → Webview): `{ kind: "event", event: "runStarted" | "progress" | "runCompleted" | "runFailed" | "runCancelled", ...payload }`

Validation: every `request.id` must receive exactly one `response` with the same `id` (correlation); commands are a closed enum.

## Entity: Extension Settings (manifest `contributes.configuration`)

See `contracts/extension-manifest.md`.

- `ppi.profile`: enum `["python","odoo"]`, default `"python"`, scope Workspace+Global.
- `ppi.analysisDir`: string, default `""` (CLI default), scope Workspace+Global.
- `ppi.pythonExecutable`: string, default `""`.
- `ppi.cliPath`: string, default `""`.

## Relationships

```
Workspace Profile --resolves--> CLI invocation args
Analysis Run --spawns--> `ppi analyze --json` --emits--> ProgressEvent*
Dashboard View --owns--> QueryBridge --spawns--> `ppi rpc` servant
Webview --postMessage(request)--> extension --JSON-RPC--> `ppi rpc` --dispatcher--> StoreReader (read-only)
```

No entity in the extension references DuckDB, the store schema, or analysis internals directly — all such access is delegated to the CLI (Principle V, FR-015).
